## 2. 专项：云端采集失败与「廉价代理」（大问题）

### 2.1 为什么这是大问题

GitHub Actions 跑在 **数据中心 IP** 上。X 对这类 IP 常：

- 直接拒绝 / 403 / 空结果  
- 要求额外验证  
- 导致 **Cookie 有效但在云端仍拉不到**

Agent-Reach 文档也写明：本地往往免费可用；**服务器环境可能需要代理**（他们举例约 $1/月量级，实际因供应商而异）。

因此：**「关机 + 云端采 X + 免费 Cookie」三者同时成立时，代理/网络路径往往成为成败关键。**

### 2.2 可选策略（按推荐顺序）

#### 策略 P0 — 先测再买（默认第一步）

1. 本机 Cookie 采 X（阶段 0）  
2. **同一 Cookie 丢进 Actions 试 3～7 天，不加代理**  
3. 统计：成功率、空结果率、是否秒封 Cookie  

| 结果 | 下一步 |
|------|--------|
| 成功率高（如 ≥5/7 天有足够推文） | **暂不买代理**，只做告警 + Cookie 轮换文档 |
| 经常空/403 | 进入 P1 廉价代理，或 P2 混合 |

**原则：代理是「被验证需要」再上，不是一上来必选项。**

---

#### 策略 P1 — 廉价住宅/ISP 代理（云端仍关机）

当 Actions 直连失败时：

| 类型 | 大致月费体感 | 说明 |
|------|----------------|------|
| 共享住宅代理（按流量） | 常 **几美元～十余美元/月** 起 | 个人日更流量很小，选最低档往往够 |
| 数据中心代理 | 更便宜但不一定骗过 X | 可能白买 |
| 机场/订阅里的「家宽落地」 | 看你是否已有 | 若协议允许给脚本用 HTTP(S) 代理可试 |

**技术接法（计划态）：**

```text
Actions 环境变量：
  HTTPS_PROXY / HTTP_PROXY
  或 twitter-cli / 底层请求支持的 proxy 参数
Cookie 仍用你的 TWITTER_* Secrets
```

**运维：**

- 代理账号与 Cookie 一样放 Secrets  
- 代理挂了 → Actions 失败 → GitHub 邮件通知你  
- 文档写清：换代理商 / 换套餐的步骤  

**风险：**  

- 「廉价」不等于永远 $1；被滥用的共享池可能同样被 X 拉黑  
- 需要你接受 **小额月费**（若坚持 0 元且云端采 X，可能做不到稳态）

---

#### 策略 P2 — 混合架构（代理仍贵或不想用时）

```text
采集 X  →  放在「便宜常开的小机器 / 家宽」上（仅凌晨跑 5 分钟）
总结 + 建站 + Pages → 仍可 GitHub Actions
```

| 优点 | 缺点 |
|------|------|
| 家宽 IP 更像真人，少代理费 | **不完全「电脑全关」**；或要一台 24h 低配 VPS/旧手机/旁路设备 |
| Cookie 更稳 | 多一个运行环境 |

「廉价」可以是：

- 闲置安卓 + 定时 Tasker（进阶）  
- 极低配 VPS **仅当采集机**（若机房 IP 仍废，则 VPS 也要上代理，又回到 P1）  
- 仅当本机「每天早上自己会开机」时用计划任务采 X（你已表示倾向关机，故 P2 为备选）

---

#### 策略 P3 — 不推荐当主方案

| 方案 | 原因 |
|------|------|
| 随机免费公开代理 | 极不稳、不安全（Cookie 泄露风险） |
| 多号轮换硬刚 | 个人项目过重，且仍可能违规 |
| 再买 X 官方 API | 与你的约束冲突 |

### 2.3 代理问题的决策树（写入计划）

```text
本机 Cookie 采集 OK？
  ├─ 否 → 先修 Cookie/CLI，不要谈云端
  └─ 是 → Actions 直连 3～7 天
            ├─ 成功率可接受 → 不上代理，进入网站阶段
            ├─ 失败为主 → 评估廉价住宅代理（P1）
            │              ├─ 可接受月费 → 上代理，关机全自动
            │              └─ 不能接受月费 → P2 混合 / 或降低「关机」要求
            └─ Cookie 被封 → 降低频率、减少请求、更新 Cookie 手册
```

### 2.4 对「廉价代理是大问题」的产品态度

- **承认：** 这是本方案最大不确定性，不是细节。  
- **缓解：** 分阶段验证（P0 先测）；代理按需上；失败可观测。  
- **不能承诺：** 「永远 $0 且 100% 云端采 X 成功」。  
- **可承诺：** 架构上预留 `PROXY` 配置；文档写清失败时你怎么修。  

**建议你在实现前再表态一次（可后补）：**

- **A1** 先 P0 测满一周，再决定买不买代理  
- **A2** 直接预留代理预算（你接受大概每月上限：？元）  
- **A3** 更接受 P2（采集不放在 GitHub 机房）

## 3. 专项：如何让工具读到「你想要的内容」（大问题）

Agent-Reach 只解决 **「能读 X」**，不解决 **「读得对不对胃口」**。  
「想要的内容」必须用 **配置 + 策略 + 反馈** 产品化。

### 3.1 内容画像（建议你填空，实现时写进配置）

| 维度 | 问题 | 示例 |
|------|------|------|
| 必看账号 | 哪些 @ 一天不看就难受？ | OpenAI, Anthropic, xai, … |
| 可选账号 | 有空再看 | 媒体号、KOL |
| 主题关键词 | 中英文各 5～15 个 | agent, RAG, LLM release, 开源… |
| 降权/拉黑 | 永远不想进日报的 | 纯币圈喊单、饭圈 |
| 语言 | 中/英/都要 | 中文为主，原文可英 |
| 视角 | 总结时的读者 | CS / agent 工程 / side hustle |
| 条数 | 每天几条 | 6～10 |
| 关注优先 | 是否强绑定 Following 时间线 | 是 / 否 |

**没有这份画像，任何采集工具都会「能读但读不对」。**

---

### 3.2 采集策略设计（计划中的机制）

用配置文件（如 `config/accounts.yaml`）表达，而不是写死在代码里：

```text
1) Following 时间线（可选，最贴近「我关注的」）
2) 指定账号 user timeline（必看列表，可配置）
3) 关键词 search Latest（补漏热点）
4) 合并去重 → 截断到 N 条 → 交给 DeepSeek
```

| 策略 | 作用 |
|------|------|
| **白名单账号** | 保证大佬/工具官方不丢 |
| **搜索词** | 抓「你关心的主题」而非全站热搜 |
| **黑名单词/账号** | 过滤噪音 |
| **条数上限** | 控制 DeepSeek 费用与噪音 |
| **来源标记** | 方便你事后说「这条不该来」 |

DeepSeek 侧：

- System prompt 固定你的 CS/agent 视角  
- **只根据输入推文写**，禁止编造  
- 输出完整 ITEM 结构，便于上站  

---

### 3.3 「读得不对」时怎么迭代（产品闭环）

| 现象 | 调整什么 |
|------|----------|
| 大佬动态总缺 | 加 `accounts` 白名单 |
| 太杂、币圈多 | 加黑名单；收紧 search |
| 没有「我关注的」味道 | 打开/加重 following feed |
| 总结偏题 | 改 DeepSeek prompt，不先怪采集 |
| 条数虚高胡编 | 检查 raw JSON 是否太空；空则不要硬总结 |

**建议上线后每周看 1 次站点 + raw，用改 yaml/prompt 迭代，而不是改架构。**

---

### 3.4 和「Grok 推送」的差别（预期管理）

| Grok Tasks | 本方案 |
|------------|--------|
| Grok 自己决定搜什么 + 怎么写 | **你配置**采什么 + DeepSeek 按规则写 |
| 文风偏 Grok | 文风偏 DeepSeek（可用 prompt 贴近） |
| 邮件截断 | **全文上站** |

停用 Tasks 后，**质量取决于配置是否像「你的关注与口味」**，需要你前期花一点时间填账号/关键词。
